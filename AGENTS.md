# em-conta Agent Notes

## Project Shape

- Use `pnpm` only; root `package.json` pins `pnpm@10.33.0` and Node `>=22.12.0 <25`.
- Monorepo packages are `apps/api`, `apps/web`, and `packages/dto`; path aliases are `@api/*`, `@web/*`, and `@dto/*`.
- Shared request/response contracts live in `packages/dto/src` and use Valibot; API controllers import these DTO schemas for boundary validation.

## Commands

- Full local dev: `pnpm dev` runs web Vite and API Wrangler concurrently.
- API dev/deploy/typegen: `pnpm api:dev`, `pnpm api:deploy`, `pnpm api:typegen`.
- Web dev/build/preview/deploy: `pnpm web:dev`, `pnpm web:build`, `pnpm web:preview`, `pnpm web:deploy`.
- Lint/format both apps: `pnpm lint`, `pnpm lint:fix`, `pnpm format`, `pnpm format:check`.
- API tests only: `pnpm api:test`; focused Vitest examples: `pnpm --filter api test -- test/index.spec.ts` or `pnpm --filter api test -- -t "health"`.
- There is no root `test` or `typecheck` script; use the package scripts that exist instead of inventing one.

## Delivery Workflow

- Durable MVP sequencing lives in `docs/mvp-roadmap.md`; update slice status only after code, validation, and review fixes are done.
- Main builder owns orchestration and coding. Use `accounting-advisor` before changing journal entries, reports, balances, DRE/BP/liquidity, dashboard numbers, receivables, payables, or inventory accounting.
- Use `code-reviewer` as a read-only review pass before finalizing a PR-sized slice; fix clear in-scope findings and rerun focused checks.
- OpenCode workflow commands: `/plan-slice` for planning only, `/next-pr` for implementation + review loop, `/review-pr` for standalone diff review/fix, `/validate-mvp` for demo acceptance checks.
- Keep work in small vertical slices: backend contract/rules first, focused tests or checks, then frontend integration.

## API App

- Cloudflare Worker entry is `apps/api/src/index.ts`; Hono app wiring is `apps/api/src/app.ts` under `/api/v1` plus `/api/auth/*` for Better Auth.
- `apps/api/src/runtime.ts` registers Effect layers. Add new repos/services there or `c.env.runtime.runPromise(...)` calls will fail at runtime.
- Feature folders under `apps/api/src/features/*` follow controller -> service -> repo: controllers stay thin, services hold business rules, repos wrap Drizzle persistence.
- Repos use `Database.execute`; multi-write flows that must be atomic use Drizzle transactions inside a repo, as in sales posting.
- Expected business failures use tagged Effect errors (`Data.TaggedError`) and controllers map them to JSON error codes.

## Database

- Drizzle schema files are in `apps/api/src/db/schema/*.ts`; generated migrations go in `apps/api/src/db/migrations`.
- Generate and run migrations with `pnpm api:db:generate` then `pnpm api:db:migrate`.
- Drizzle config loads `dotenv/config` and needs `CLOUDFLARE_HYPERDRIVE_LOCAL_CONNECTION_STRING_HYPERDRIVE` set, usually from `apps/api/.env`.
- Wrangler binds Hyperdrive as `HYPERDRIVE` with local Postgres `postgresql://postgres:root@localhost:5432/postgres`; local DB commands assume that service exists.
- Run `pnpm api:typegen` after changing `apps/api/wrangler.jsonc` bindings or environment shape.

## Web App

- React entry is `apps/web/src/main.tsx`; file routes live in `apps/web/src/routes` and feature UI/query code lives in `apps/web/src/features`.
- `apps/web/src/routeTree.gen.ts` is generated by TanStack Router; do not hand-edit it.
- API calls should use the typed Hono client in `apps/web/src/lib/api.ts` and the `callApi` neverthrow wrapper, not raw `fetch`.
- Web runtime needs `VITE_API_URL`; auth and API clients both read it from Vite env.
- List/detail workflows should use the table-first UX: `@tanstack/react-table` tables with search, useful filters, and row click opening a detail modal. Keep create/edit forms in modals launched by top-page action buttons instead of large inline forms.
- Sales must keep a clear shortcut to product/service creation from the sales screen so missing catalog items can be added before selling.

## Accounting Domain Constraints

- Automatic postings must resolve accounts by stable `AccountKey` values (`cash`, `bank_checking`, `accounts_receivable`, `inventory`, `sales_revenue`, `services_revenue`, `cogs`, `taxes_payable`), not by display names.
- Sales, receipts, stock movements, and journal entries are already implemented; preserve source links and balanced debit/credit journal lines.
- Product sales that track inventory snapshot costs at posting time and must not recompute old journal/stock movement amounts later.

## Tooling Notes

- Biome is the formatter/linter for both apps, with 2-space indentation and 100-column width.
- API tests run through `@cloudflare/vitest-pool-workers` using `apps/api/wrangler.jsonc`; failures can involve Worker/Wrangler config, not just application code.
